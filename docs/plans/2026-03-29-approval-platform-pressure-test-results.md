## 压测结果判断

### 读路径

- 主要瓶颈仍然集中在：
  - `tasks.page`
  - `approval-sheets.page`
- 低风险分页优化后，`approval-sheets.page` 在 50 并发下比旧基线更稳定，但 100 并发仍然出现失败
- `tasks.page` 在高并发下仍是最重入口之一，说明当前瓶颈不只是“全量 enrich”，还包括任务查询、变量读取和流程定义解析等多步聚合

### 写路径

- `oa.leaves.create`
- `tasks.claim`
- `tasks.complete`

这三条写路径在 `10 / 50 / 100` 并发下都保持稳定，没有出现错误率抬升，当前不是主要性能瓶颈

## 瓶颈判断

综合本轮压测数据，当前瓶颈优先级如下：

1. `审批单分页 / 任务分页查询`
   - 读路径入口最重
   - 高并发下最先暴露超时和失败
2. `审批详情及关联数据聚合`
   - 虽然单次详情较快，但在列表联动、详情联动、统计联动时可能形成放大效应
3. `数据库读压力与索引命中`
   - 需要重点排查分页查询、关联查询、排序字段和过滤条件
4. `服务端线程与连接资源`
   - 在 100 并发下 `approval-sheets.page` 已出现失败，除了 SQL 本身，还需要关注线程池、连接池和 Flowable 服务调用放大

## 优化优先级

### P0

- 优化 `tasks.page`
- 优化 `approval-sheets.page`
- 排查 100 并发下失败和超时根因
- 补充关键表和常用过滤字段索引，并核对 Flowable 查询路径

### P1

- 优化 `tasks.detail` 的关联查询和投影字段
- 减少列表页和详情页的重复查询
- 评估缓存或预聚合是否能缓解高频只读场景

### P2

- 对写路径继续做稳定性回归
- 增强监控与慢查询采样
- 将压测结果纳入回归基线

## 下一轮压测建议

建议下一轮按以下顺序推进：

1. 先做 SQL / 索引层分析
   - 重点看 `tasks.page` 和 `approval-sheets.page`
   - 输出慢查询、执行计划、索引命中率

2. 做 20 / 30 / 50 并发阶梯压测
   - 观察从 10 并发到 50 并发的退化拐点
   - 确认是否存在明显的阈值区间

3. 拆分读写压测
   - 只压列表
   - 只压详情
   - 只压审批单分页
   - 只压写动作

4. 加入服务端指标采样
   - 请求耗时分布
   - DB 慢查询
   - 连接池占用
   - 线程池排队情况

5. 再做 50 / 100 并发复测
   - 验证优化前后差异
   - 以 `approval-sheets.page` 错误率是否归零、读接口 `p95` 是否显著下降作为主要判定

## 结论

当前审批平台的写路径已经稳定，但读路径在 `50+` 并发下明显退化，在 `100` 并发下 `approval-sheets.page` 已出现真实失败。  
本轮完成了正式压测和一轮低风险性能收口，但还不能说“性能问题已解决”。下一步应优先深挖 `tasks.page` 和 `approval-sheets.page` 的真实热点，再继续更高并发或更复杂的上线压测。
