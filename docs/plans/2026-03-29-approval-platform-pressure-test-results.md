# 审批平台压测结果报告

## 测试范围

本轮压测聚焦审批平台的核心读写路径，验证在不同并发下的响应时间与稳定性：

- `tasks.page`
- `approval-sheets.page`
- `tasks.detail`
- `oa.leaves.create`
- `tasks.claim`
- `tasks.complete`

其中：

- 读路径重点关注列表、详情、审批单分页接口
- 写路径重点关注请假发起、认领、同意等核心动作

## 测试环境

- 仓库：`/Users/west/dev/code/west/west-flow-ai`
- 压测对象：当前本地审批平台服务
- 运行环境：本机开发环境
- 并发方式：按接口分别做 10 并发基线与 50 / 100 并发读压测
- 判定依据：
  - 平均响应时间
  - `p95`
  - 最大响应时间
  - 是否出现 `ReadTimeout`

## 压测结果判断

### 读路径

50 / 100 并发读压测均发生 `ReadTimeout`，说明当前读路径在高并发下不稳定。

从已有数据看，问题主要集中在：

- `tasks.page`
- `approval-sheets.page`

这两条路径在 10 并发下已经出现 1s 级别平均耗时，继续放大并发后更容易触发超时。

### 写路径

目前已知基线中：

- `oa.leaves.create`
- `tasks.claim`
- `tasks.complete`

表现明显优于读路径，短期内不是主要瓶颈。

## 瓶颈判断

综合基线与高并发超时现象，当前瓶颈优先级如下：

1. `审批单分页 / 任务分页查询`
   - 读路径入口最重
   - 高并发下最先暴露超时
2. `审批详情及关联数据聚合`
   - 虽然单次详情较快，但在列表联动、详情联动、统计联动时可能形成放大效应
3. `数据库读压力与索引命中`
   - 需要重点排查分页查询、关联查询、排序字段和过滤条件
4. `服务端线程与连接资源`
   - ReadTimeout 说明除了 SQL 本身，还可能存在线程池、连接池或上游等待时间偏长的问题

## 优化优先级

### P0

- 优化 `tasks.page`
- 优化 `approval-sheets.page`
- 排查高并发下的 `ReadTimeout` 根因
- 补充关键表和常用过滤字段索引

### P1

- 优化 `tasks.detail` 的关联查询和投影字段
- 减少列表页和详情页的重复查询
- 评估缓存或预聚合是否能缓解高频只读场景

### P2

- 对写路径继续做稳定性回归
- 增强监控与慢查询采样
- 将压测结果纳入回归基线

## 下一轮压测建议

建议下一轮按以下顺序推进：

1. 先做 SQL / 索引层分析
   - 重点看 `tasks.page` 和 `approval-sheets.page`
   - 输出慢查询、执行计划、索引命中率

2. 做 20 / 30 / 50 并发阶梯压测
   - 观察从 10 并发到 50 并发的退化拐点
   - 确认是否存在明显的阈值区间

3. 拆分读写压测
   - 只压列表
   - 只压详情
   - 只压审批单分页
   - 只压写动作

4. 加入服务端指标采样
   - 请求耗时分布
   - DB 慢查询
   - 连接池占用
   - 线程池排队情况

5. 再做 50 / 100 并发复测
   - 验证优化前后差异
   - 以 `ReadTimeout` 是否消失作为主要判定

## 结论

当前审批平台的写路径基本可用，但读路径在 50+ 并发下已经不稳定。  
下一步应优先处理列表和审批单分页查询，先把高频读接口从 1s 级别压下来，再考虑更高并发和更复杂的功能扩展。
