# 剩余任务执行顺序设计

> 状态：Approved v1
> 日期：2026-03-22
> 适用范围：`codex/m0-foundation` 后续所有开发批次

## 1. 背景

当前仓库已经完成以下基础闭环：

- 系统管理核心模块：用户、角色、菜单、公司、部门、岗位、代理关系、离职转办、通知渠道、触发器
- 流程平台基础：流程定义、流程设计、DSL 持久化、发布、OA 发起、流程中心、审批单详情
- 运行态动作基础：认领、转办、退回、驳回、跳转、拿回、唤醒、加签、减签、撤销、催办、已阅、委派、代理、离职转办
- 编排自动化基础：超时审批、自动提醒、定时节点、触发节点的设计器配置、通知发送、系统配置页

但当前代码仍存在两个核心事实：

- 运行态仍以 `ProcessDemoService` 为中心，尚未切换到真实 Flowable 运行时
- 后续任务需要强制按顺序推进，避免再出现“横向铺功能但底层仍是 demo”的情况

本设计用于冻结剩余任务的执行顺序。

## 2. 总原则

- 后续开发严格按本文件定义的 6 个阶段推进
- 前一阶段未收口，不进入下一阶段的大规模实现
- 每一阶段开始前，都允许再补一份该阶段专用设计文档与实施计划
- 不再按“想到什么做什么”的方式横向扩散模块

## 3. 固定执行顺序

### Phase 1：全仓中文注释治理

目标：

- 前后端业务代码中的类、方法都补齐简单明了的中文注释
- 优先覆盖领域服务、控制器、核心组件、复杂状态机、流程转换与运行态逻辑

本阶段重点：

- 类注释
- 方法注释
- 复杂分支注释
- 权限判断注释
- 流程节点与动作语义注释

明确约束：

- 注释必须解释业务意图，不写废话注释
- `record`、简单 DTO、纯常量类不强制写大段注释，但公开方法和复杂逻辑仍要补

### Phase 2：系统管理补齐与清理

目标：

- 补齐系统管理剩余模块
- 清理 `shadcn-admin` 残留无关页面
- 在系统管理下补齐“通知管理”和“消息管理”

本阶段重点：

- 字典管理
- 日志管理
- 监控管理
- 文件管理
- 通知管理
- 消息管理
- 清理 `apps / chats / tasks / help-center` 等残留页面与菜单

说明：

- “通知渠道配置”与“触发器管理”已具备基础，此阶段补的是更完整的运营后台能力
- “消息管理”指站内消息、消息记录、消息状态、消息查询等业务后台，而不是单纯渠道配置

### Phase 3：真运行态

目标：

- 将当前 `demo` 运行态替换为真实流程运行时
- 让流程实例、任务、动作、轨迹、事件真正建立在 Flowable 运行模型之上

本阶段重点：

- 替换 `ProcessDemoService`
- 去掉 `/process-runtime/demo/*` 语义
- 真实实例与任务持久化
- 真实事件链路
- 真实动作权限与状态校验
- 真实定时扫描与自动推进

这是平台后续能否稳定扩展的关键阶段。

### Phase 4：流程管理后台

目标：

- 把流程管理应有的后台页面补齐成完整平台侧入口

本阶段重点：

- 流程版本
- 流程发布记录
- 流程实例监控
- 流程操作日志
- 审批意见配置
- 业务流程绑定

说明：

- 这一阶段聚焦平台管理后台，不再重复建设运行态工作台
- OA 查询与流程中心继续共用统一运行态

### Phase 5：高级审批能力

目标：

- 在真实运行态基础上继续补齐更复杂的中国式审批能力

本阶段重点：

- 主子流程
- 追加
- 动态构建
- 包容分支
- 终止
- 沟通
- 穿越时空
- 真正的顺序会签、并行会签、或签、票签运行语义
- 更完整的 SLA、提醒、调度、策略能力

说明：

- 本阶段不再接受“只有 DSL 配置，没有运行闭环”的半成品

### Phase 6：AI + PLM

目标：

- 在平台主链路稳定后，再进入 AI 和 PLM 两条扩展线

本阶段重点：

- AI Copilot 前端入口与对话历史
- AI 权限与工具调用控制
- 流程设计智能体
- 智能填报
- AI 发起流程
- AI 处理待办
- AI 统计问答
- PLM 实际业务案例

说明：

- `Spring AI`、`aiCapabilities`、`MinIO` 等基础依赖已在仓库中预埋，但不等于 AI 已完成
- PLM 不再只停留在“后续扩展”口径，需要在本阶段形成真实样例

## 4. 阶段边界

为了避免顺序被打乱，后续执行必须满足以下边界：

- Phase 2 不抢跑真实运行态替换
- Phase 3 不夹带 AI Copilot
- Phase 4 不重复建设 OA 运行态页
- Phase 5 不接受仅文档化、不接受仅按钮化
- Phase 6 建立在前 5 阶段稳定基础上

## 5. 当前推荐下一步

下一步应先进入：

- `Phase 1：全仓中文注释治理`

在注释治理阶段收口后，再进入：

- `Phase 2：系统管理补齐与清理`

这两步结束后，才正式开启：

- `Phase 3：真运行态`
