# 运行态闭环与基础改造设计

> 状态：Frozen v1
> 日期：2026-03-22
> 范围：`codex/m0-foundation`

## 1. 目标

本轮在已有“委派 / 代理 / 离职转办”运行态能力基础上，继续收口成完整可交付闭环，并同步完成两项基础改造：

- 完成协作动作的系统侧闭环，而不再停留在固定规则和运行态入口
- 对前后端手写业务代码补充简单明了的中文注释
- 后端统一升级到 `JDK 21`
- 后端代码采用“`record` 优先、`Lombok` 选择性使用”的统一准则

## 2. 本轮范围

### 2.1 协作动作闭环

本轮补齐以下能力：

- 代理关系管理
- 离职转办执行页
- 委派 / 代理 / 离职转办在流程中心与审批单详情中的完整展示
- 运行态权限与页面权限联动

### 2.2 代码治理

本轮同步治理：

- `frontend/src` 下手写业务代码
- `backend/src/main/java` 与 `backend/src/test/java` 下手写代码

### 2.3 明确排除项

以下内容本轮不做：

- 自动代理生效时间窗
- 委派撤回
- 离职转办批量计划任务
- 把 `record` DTO 反向重构为 Lombok 类
- 对自动生成文件、静态 JSON、纯素材组件做机械式注释灌水

## 3. 协作动作闭环设计

### 3.1 菜单与页面

系统侧新增独立中文页面，不与其他模块混页签：

- `系统管理 / 代理关系管理`
- `系统管理 / 离职转办执行`

流程中心继续保留运行态入口：

- 审批单详情页支持 `委派`
- 待办列表工具栏支持 `离职转办`
- 代理关系生效后，代理人可在待办直接看到可代理处理的任务

### 3.2 代理关系管理

页面能力：

- 列表页
- 详情页
- 新建页
- 编辑页

字段最小集合：

- 委托人用户
- 代理人用户
- 状态
- 生效说明

本期先不做复杂时间窗，只做：

- `ACTIVE`
- `DISABLED`

### 3.3 离职转办执行页

页面定位：

- 不是普通 CRUD
- 是管理动作执行页
- 仍然使用独立列表页 + 执行确认页模式

能力要求：

- 输入来源用户
- 输入目标用户
- 预览可迁移任务数
- 确认执行离职转办
- 展示执行结果与任务明细

### 3.4 权限规则

- 代理关系管理页：仅流程管理员可见
- 离职转办执行页：仅流程管理员可见
- 委派：当前任务直接办理人可执行
- 代理代办：仅激活代理关系下的代理人可执行
- 离职转办：仅流程管理员可执行

## 4. 中文注释治理规则

### 4.1 覆盖边界

需要覆盖的代码：

- `frontend/src/**/*.ts`
- `frontend/src/**/*.tsx`
- `backend/src/main/java/**/*.java`
- `backend/src/test/java/**/*.java`

排除：

- `frontend/src/routeTree.gen.ts`
- `frontend/src/vite-env.d.ts`
- `frontend/src/tanstack-table.d.ts`
- `frontend/src/**/*.json`
- 纯静态图标、纯 SVG 包装组件

### 4.2 注释标准

注释必须满足：

- 中文
- 简单明了
- 说明“为什么”和“这段逻辑在做什么”
- 不写逐行废话

最小覆盖要求：

- 每个手写业务文件至少有一处中文说明
- 复杂状态机、分支决策、流程路由、表格协议、权限判断必须有中文注释
- 测试文件需要用中文说明用例意图

禁止：

- “给变量赋值”这类废话注释
- 把英文注释原样机翻
- 只为了满足数量硬塞无意义注释

## 5. record 与 Lombok 准则

### 5.1 record 优先

以下场景优先使用 `record`：

- Request / Response DTO
- 不可变查询对象
- 只承载数据、无可变生命周期的值对象
- mapper 返回模型

### 5.2 Lombok 适用场景

以下场景适合使用 Lombok：

- 可变内部状态模型
- 需要大量 getter / setter / constructor 的实体类
- 需要 builder 的复杂装配对象
- 典型 Spring 配置类或手写 POJO

推荐注解：

- `@Getter`
- `@Setter`
- `@RequiredArgsConstructor`
- `@NoArgsConstructor`
- `@AllArgsConstructor`
- `@Builder`

### 5.3 禁止事项

- 不为了“使用 Lombok”而把 `record` 改回 class
- 不在实体上滥用 `@Data`
- 不对包含复杂业务方法的类机械套 Lombok 后再降低可读性

## 6. JDK 21 升级设计

本轮统一：

- `backend/pom.xml` 的 `java.version` 改为 `21`
- 编译、测试、文档与脚本统一到 `JDK 21`

升级要求：

- 不引入与 `Java 17` 绑定的旧写法
- 保持现有测试全绿
- 如需新增编译插件配置，只做最小必要改动

## 7. 多代理拆分

### Agent A：后端协作配置闭环

- 代理关系管理 API
- 离职转办执行与预览 API
- 运行态权限对接

### Agent B：前端系统管理页面

- 代理关系管理列表 / 详情 / 新建 / 编辑
- 离职转办执行页
- 侧边栏与路由接入

### Agent C：全仓中文注释治理

- 优先处理本轮相关模块
- 再扩展到其余手写业务代码
- 只补高价值中文注释

### Agent D：JDK 21 与 Lombok 改造

- `pom.xml`
- Lombok 引入与最小重构
- 不破坏 `record` 体系

## 8. 验收标准

- 系统管理存在独立“代理关系管理”和“离职转办执行”页面
- 代理关系不再只依赖 fixture 硬编码入口，能通过页面维护
- 流程中心、审批单详情、系统管理三侧语义一致
- 前后端手写业务代码完成中文注释治理
- 后端统一到 `JDK 21`
- Lombok 使用符合“record 优先”准则
- 全量测试通过
