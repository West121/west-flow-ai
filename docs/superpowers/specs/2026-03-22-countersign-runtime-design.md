# Phase 5 会签策略真实闭环设计

> 状态：Approved v1
> 日期：2026-03-22
> 适用范围：`codex/m0-foundation` Phase 5 第一批

## 1. 背景

当前平台已经完成：

- 真实 `Flowable BPMN` 运行态
- 流程定义、设计、发布、版本、实例监控、操作日志等后台能力
- 单人审批主链路与常见中国式动作

但审批节点的“多人处理策略”仍未形成真实闭环：

- 设计器里已有会签模式选项和部分 DSL 校验
- 运行态仍以单人节点和平台补充动作语义为主
- 审批单详情、流程图和轨迹没有完整表达会签组、成员、权重、阈值和自动结束原因

Phase 5 第一批不继续横向扩展主子流程、追加、动态构建等能力，而是先把最核心的多人审批策略做成真实 `Flowable` 闭环。

## 2. 目标

本阶段完成后，平台需要满足以下目标：

- 审批节点支持 `顺序会签 / 并行会签 / 或签 / 票签`
- 设计器配置、DSL、BPMN/运行态映射、任务推进、详情展示、轨迹留痕全部打通
- 所有会签策略建立在真实 `Flowable` 多实例用户任务之上，不再发明第二套执行引擎
- 驳回、退回、跳转后重新进入会签节点时，能按明确策略重建或恢复会签组
- 审批单详情、流程图、任务轨迹能显示会签组与成员级状态

## 4. 产品语义

### 4.1 顺序会签

- 同一审批节点设置多位处理人
- 按配置顺序依次生成活动任务
- 前一人完成后，下一人才会收到任务
- 全部成员完成后进入下一步

### 4.2 并行会签

- 同一审批节点设置多位处理人
- 所有成员同时生成活动任务
- 所有成员都完成后，流程才进入下一步

### 4.3 或签

- 同一审批节点设置多位处理人
- 所有成员同时生成活动任务
- 任意一人审批通过即可进入下一步
- 剩余未处理同组任务自动结束并保留轨迹

### 4.4 票签

- 同一审批节点设置多位处理人及各自权重
- 所有成员同时生成活动任务
- 每位成员处理后累计通过票权重或拒绝票权重
- 达到阈值后立即决议：
  - 达到通过阈值：节点通过
  - 达到拒绝阈值：节点拒绝
- 剩余未处理同组任务自动结束并保留轨迹

## 5. DSL 与设计器

### 5.1 审批节点新增字段

审批节点在现有配置基础上补以下字段：

- `approvalMode`
  - `SINGLE`
  - `SEQUENTIAL`
  - `PARALLEL`
  - `OR_SIGN`
  - `VOTE`
- `assignees`
  - 处理人列表
- `voteRule`
  - `thresholdPercent`
  - `passCondition`
  - `rejectCondition`
  - `weights`
- `reapprovePolicy`
  - `RESTART_ALL`
  - `CONTINUE_PROGRESS`
- `autoFinishRemaining`
  - `true / false`

### 5.2 校验规则

- `SEQUENTIAL / PARALLEL / OR_SIGN / VOTE` 至少配置 2 名处理人
- `VOTE` 必须配置票签阈值
- `VOTE` 每位处理人必须存在权重，且总权重必须大于 0
- `OR_SIGN` 必须启用 `autoFinishRemaining`
- 非 `VOTE` 模式不允许带票签权重

### 5.3 设计器交互

设计器审批节点属性面板增加：

- 会签模式选择
- 顺序列表或成员列表
- 票签阈值与权重配置
- 重新审批策略
- 自动结束剩余任务开关

## 6. Flowable 实现策略

### 6.1 总原则

- 顺序与并行会签直接基于 `Flowable` 多实例用户任务
- `或签` 和 `票签` 也基于多实例用户任务，但平台层负责补充完成条件聚合与剩余任务自动结束
- 不在控制器里写会签语义
- 不把复杂票签条件塞满 BPMN 表达式

### 6.2 顺序会签

- 使用串行多实例用户任务
- `isSequential=true`
- 每次仅一个成员任务处于活动状态
- 上一个完成后自动生成下一个

### 6.3 并行会签

- 使用并行多实例用户任务
- 所有成员任务同时激活
- 全部完成后推进到下一节点

### 6.4 或签

- 使用并行多实例用户任务
- 任意一人通过后，平台层统一结束剩余同组任务
- 自动结束任务必须写动作轨迹，原因标记为 `AUTO_FINISHED_BY_OR_SIGN`

### 6.5 票签

- 使用并行多实例用户任务
- 每次成员处理后更新票签聚合快照
- 达到通过/拒绝阈值后：
  - 结束剩余同组任务
  - 写票签决议事件
  - 推进或拒绝当前节点

## 7. 后端架构

### 7.1 新增服务

- `FlowableCountersignService`
  - 初始化会签组
  - 聚合成员结果
  - 判断推进条件
  - 结束剩余任务
  - 写会签轨迹快照

### 7.2 职责边界

- `flowable helper/facade`
  - 只负责底层引擎操作
- `FlowableCountersignService`
  - 负责会签节点运行语义
- `FlowableProcessRuntimeService`
  - 继续作为任务详情、动作入口与审批单组装协调层

## 8. 平台扩展表

为避免完全依赖 Flowable 历史表硬拼展示语义，新增 3 类平台表：

- `wf_task_group`
  - 一次进入会签节点生成一个任务组
- `wf_task_group_member`
  - 会签成员任务、顺序号、权重、状态、处理结果
- `wf_task_vote_snapshot`
  - 票签累计权重、通过比例、拒绝比例、命中阈值时间

这些表只承载平台产品语义，不复制引擎全量状态。

## 9. 接口影响

保持现有 `/api/v1/process-runtime/*` 路径，不新增会签专用控制器。

增强现有接口：

- `GET /tasks/{taskId}`
  - 增加会签组、成员状态、顺序号、权重、累计票数
- `GET /approval-sheets/by-business`
  - 增加节点级会签摘要与轨迹
- `POST /tasks/{taskId}/complete`
  - 内部按 `approvalMode` 和会签聚合状态决定是否真正推进

新增只读接口：

- `GET /instances/{instanceId}/task-groups`
  - 用于审批单详情和流程图侧栏展示会签进度
