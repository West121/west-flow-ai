# 流程设计器重构与高级结构能力设计

> 状态：Draft
> 日期：2026-03-23
> 适用范围：`main` 下一阶段流程设计器重构与高级结构能力深化

## 1. 背景

当前平台已经具备：

- 真实 `Flowable BPMN` 运行态
- 流程设计器、DSL、发布、流程管理后台
- 主子流程、追加/动态构建、包容分支、沟通、穿越时空、终止高级策略的阶段性闭环

但当前设计器和高级结构能力还有两个明显问题：

1. **设计器交互效率低**
   - 流程属性和节点属性上下串行堆叠，页面过长
   - 画布区域不够聚焦，一屏看不全
   - 节点选人仍偏“编码输入”，业务不可用
   - 缺少表单字段选人、自定义公式配置

2. **高级结构能力仍缺最后一层深化**
   - 更深主子流程编排还没统一落点
   - 动态构建规则仍偏模板化，不够结构化
   - 包容分支当前只完成基础命中闭环，尚未支持更复杂策略

因此这一轮需要同时处理两类工作：

- 前端设计器体验重构
- 高级结构能力下一阶段设计与代码规划

## 2. 目标

本设计覆盖两条并行主线：

1. **流程设计器重构**
   - 一屏式布局
   - 统一选人弹窗
   - 表单字段和自定义公式能力
   - 请假流程增强案例

2. **高级结构能力深化**
   - 更深主子流程编排
   - 更复杂动态构建规则
   - 更复杂包容分支策略

## 3. 设计器重构总原则

### 3.1 一屏优先

设计器必须优先保证：

- 画布始终是主视图
- 左右面板各自滚动
- 页面整体不再依赖长页面滚动来完成编辑

### 3.2 属性分层，不再纵向堆叠

属性区统一改为右侧双层结构：

- 第一层：流程属性 / 节点属性切换
- 第二层：当前对象内部按分组折叠

这样避免把流程属性和节点属性上下叠成超长表单。

### 3.3 业务优先，低频项折叠

默认只展示高频配置：

- 基本信息
- 处理人与抄送人
- 表单与字段绑定
- 路由条件
- 高级能力

低频、技术性、重复性配置默认收折，不占首屏空间。

### 3.4 React Flow 内建操作收口

缩放、适配视图、平移等操作统一回收进 `React Flow` 原生控制按钮：

- 放大
- 缩小
- 适配画布

不再保留独立重复工具栏。

## 4. 设计器目标布局

设计器改为稳定三栏：

- **左栏：节点面板**
  - 紧凑分组
  - 支持搜索
  - 保留拖拽与点击插入

- **中栏：流程画布**
  - 最大可用宽度
  - 固定撑满
  - 只在局部溢出时出现滚动

- **右栏：属性面板**
  - `Tabs`
    - `流程属性`
    - `节点属性`
  - 每个 Tab 内再用 section/card 分组
  - 面板独立滚动

移动端或窄屏时退化为：

- 画布在上
- 属性抽屉在右侧或底部弹出

但桌面主形态必须保持三栏。

## 5. 统一选人弹窗

### 5.1 目标

当前审批、抄送、加签减签、追加、动态构建等配置里存在大量：

- `userIds`
- `roleCodes`
- `departmentRef`

的手输模式，这不符合业务使用习惯。

因此统一新增一个“选人弹窗”，供所有节点复用。

### 5.2 选择来源

统一支持 3 类来源：

- 按人员
- 按角色
- 按部门

并兼容：

- 单选 / 多选
- 结构化回填
- 关键字搜索
- 标签预览

### 5.3 回填模型

弹窗不直接改 DSL，而是回填统一选择结果：

- `selectionMode`
  - `USER`
  - `ROLE`
  - `DEPARTMENT`
  - `DEPARTMENT_AND_CHILDREN`
- `selectedUsers`
- `selectedRoles`
- `selectedDepartments`

再由设计器转换成节点配置。

### 5.4 复用范围

统一选人弹窗必须覆盖：

- 审批节点
- 抄送节点
- 加签/减签配置
- 追加目标配置
- 动态构建结果配置

禁止同类配置继续分散成多种不同输入方式。

## 6. 表单字段与自定义公式

### 6.1 处理人来源扩展

审批节点选人来源扩展为：

- 指定人员
- 角色
- 部门
- 部门及子部门
- 表单字段
- 自定义公式

### 6.2 表单字段

表单字段模式用于：

- 从运行时表单字段里读取人员、部门、角色相关值
- 将字段作为审批流转判断依据

设计器需要支持：

- 字段选择器
- 字段类型显示
- 是否允许多值
- 预期映射类型提示

### 6.3 自定义公式

自定义公式不允许直接放任意脚本，先做受控表达式编辑器。

第一阶段只支持：

- 比较运算
- 布尔运算
- 字段引用
- 少量内置函数
  - `ifElse`
  - `contains`
  - `daysBetween`
  - `isBlank`

公式能力先用于两类场景：

- 审批节点按公式决定处理人
- 排他/包容网关按公式决定路由条件

### 6.4 后端约束

后端 DSL 校验需要同步补：

- 字段引用合法性
- 公式结构合法性
- 输出类型合法性

## 7. 请假流程增强案例

### 7.1 目标

用 `OA 请假审批` 作为完整演示流程，把下面能力一次串起来：

- 人员选人
- 角色选人
- 部门选人
- 表单字段
- 自定义公式
- 排他网关

### 7.2 建议流程

建议把请假流程升级为：

1. 发起人提交请假单
2. 排他网关
   - 按请假类型、请假天数分流
3. 直属主管审批
   - 按角色选人
4. 部门负责人审批
   - 按部门或部门负责人选人
5. HR 备案
   - 按角色选人
6. 总监审批
   - 按自定义公式 + 表单字段决定是否命中
7. 结束

### 7.3 价值

这条案例流程同时承担：

- 设计器体验演示
- DSL 与 BPMN 回归样例
- 业务方验证样例
- AI 后续解释流程的演示样例

## 8. 高级结构能力下一阶段

### 8.1 更深主子流程编排

在当前主子流程闭环基础上继续补：

- `callScope`
  - `CHILD_ONLY`
  - `CHILD_AND_DESCENDANTS`
- `joinMode`
  - `AUTO_RETURN`
  - `WAIT_PARENT_CONFIRM`
- `childStartStrategy`
  - `LATEST_PUBLISHED`
  - `FIXED_VERSION`
  - `SCENE_BINDING`
- `parentResumeStrategy`
  - 子流程完成后自动回父流程
  - 子流程完成后进入父流程确认点

### 8.2 更复杂动态构建规则

当前动态构建已经能生成附属任务与附属子流程，下一阶段统一扩成两类：

- `RULE_DRIVEN`
  - 根据表达式和条件规则生成
- `MODEL_DRIVEN`
  - 根据模板、场景或 AI 建议生成

规则层补充：

- 条件表达式
- 场景码匹配
- 最大构建数量
- 串行/并行策略
- 失败回退策略

### 8.3 更复杂包容分支策略

当前包容分支已完成基础命中闭环，下一阶段继续补：

- 分支优先级
- 默认分支
- 最少命中数量
- 汇聚等待策略
- 分支撤销/保留语义

并保持仍然基于 `Flowable inclusive gateway`，平台只补命中快照和决策语义。

## 9. 并行拆分

本轮推荐拆成 3 条线：

### Lane A：设计器 UI 重构

负责：

- 三栏布局
- 属性区重构
- React Flow 控制区收口
- 滚动与首屏空间优化

### Lane B：统一选人弹窗与字段/公式输入

负责：

- 统一选人弹窗
- 人员/角色/部门选择
- 表单字段选择
- 自定义公式编辑器

### Lane C：请假案例与高级结构能力深化

负责：

- 请假流程样例升级
- 排他网关案例
- 更深主子流程编排
- 更复杂动态构建规则
- 更复杂包容分支策略的后端/DSL 预留与第一批实现

## 11. 本轮不做

本轮明确不做：

- 新的可视化公式编排器
- 任意脚本执行
- AI 直接生成 BPMN 图
- 动态插入完整复杂子图
- 包容分支专属统计大盘

## 12. 建议执行顺序

1. 先完成设计器 UI 重构骨架
2. 再落统一选人弹窗与字段/公式能力
3. 再升级请假流程案例
4. 并行推进更深主子流程编排、动态构建规则、包容分支策略
