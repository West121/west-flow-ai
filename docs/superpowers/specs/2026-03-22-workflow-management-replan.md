# 流程管理重规划设计

> 状态：Approved v1
> 日期：2026-03-22
> 适用范围：`codex/m0-foundation` 后续流程平台与 OA 运行态开发

## 1. 背景

当前仓库已经具备流程定义、设计器、发布、最小运行态、认领/转办/退回等基础能力，但“流程管理”边界仍然混杂：

- `OA` 与 `流程管理` 的菜单关系不清
- 运行态入口分散，容易长出两套待办/已办
- “流程详情 / 实例详情 / 流程跟踪”被错误理解为独立菜单，而不是审批单详情页内能力
- 表单策略曾走向“独立表单管理模块”，与“表单是写好的页面”这一约束冲突

本设计用于冻结新的流程管理产品结构与实施边界。

## 2. 设计目标

- 保留 `OA` 与 `流程管理` 双入口，但不重复建设两套系统
- 统一运行态中心，待办、已办、我发起、抄送我只保留一套实现
- 将流程详情、实例详情、流程跟踪收口到审批单详情页内部
- 明确“业务发起直达业务表单”“流程中心发起先选业务入口”的双路径
- 明确“流程默认表单 + 节点覆盖表单”的代码组件表单策略
- 明确业务表、流程实例、审批意见、实例事件之间的关系模型

## 3. 产品结构

### 3.1 菜单结构

#### OA

- 请假申请
- 报销申请
- 通用申请
- OA 流程查询

#### 流程管理

- 流程中心
- 流程定义
- 流程设计
- 流程版本
- 流程发布记录
- 流程实例监控
- 流程操作日志
- 审批意见配置
- 业务流程绑定

### 3.2 运行态统一规则

`待办 / 已办 / 我发起 / 抄送我 / 发起流程` 全部归入统一的 `流程中心`。

- `OA` 下的“OA 流程查询”进入同一套流程中心页面，但默认筛选 `businessDomain=OA`
- `流程管理 > 流程中心` 进入全域流程中心，不限制业务域
- 不允许再为 `OA` 和 `流程管理` 各做一套待办/已办/我发起/抄送我

## 4. 发起规则

### 4.1 OA 业务发起

业务用户从 `OA > 请假申请 / 报销申请 / 通用申请` 直接进入业务发起页。

规则：

- 先填写业务表单
- 提交后保存业务单据
- 再根据 `businessType + sceneCode` 匹配当前生效流程绑定
- 由系统自动发起对应流程实例

### 4.2 流程中心发起

`流程中心 > 发起流程` 不是“先选流程模板”，而是“先选业务入口”。

规则：

- 用户先选业务类型
- 再跳转同一套业务发起页
- 发起逻辑仍然走业务单据保存 + 流程绑定匹配

## 5. 审批单详情页

审批单详情页是统一详情载体，不再单独做“流程详情页 / 实例详情页 / 跟踪页”菜单。

### 5.1 页面结构

- 顶部摘要区：单号、标题、状态、当前节点、发起人、发起时间、流程版本
- 业务表单区：展示业务正文，待办态可编辑当前允许字段
- 当前处理操作区：通过、拒绝、退回、转办、认领、催办、沟通、抄送等
- 审批意见区：历史意见 + 当前意见录入
- 流程图预览区：当前实例流程图与节点状态高亮
- 流程回顾区：实例执行路径动画回放
- 实例信息区：实例号、流程定义版本、业务类型、发起时间、当前状态
- 附件区：业务附件与审批附件
- 操作日志区：认领、转办、退回、催办、撤销等动作审计

### 5.2 详情页模式

- `待办` 进入可处理态
- `已办` 进入已办只读态
- `我发起` 进入发起人视角详情态
- `抄送我` 进入只读或已阅态

### 5.3 流程图预览与回顾

审批单详情页内嵌流程图，不单独跳出。

能力要求：

- 默认展示当前实例流程图
- 高亮已完成节点、当前节点、未到达节点、退回路径、转办/认领关键动作
- 支持“动画回顾”播放审批过程
- 支持暂停、继续、倍速、拖动进度
- 节点点击或悬浮时显示实例级节点执行信息

节点执行信息至少包括：

- 办理人
- 接收时间
- 阅读时间
- 办理开始时间
- 办理完成时间
- 办理时长
- 是否超时
- 操作结果
- 审批意见摘要

## 6. 表单策略

### 6.1 基本原则

- 本期不建设独立“流程表单管理”“节点表单管理”后台模块
- 表单是写好的前端代码页面/组件
- 流程表单与节点表单均在流程设计器属性面板中配置

### 6.2 生效规则

流程定义根级配置：

- `processFormKey`
- `processFormVersion`

审批节点可选覆盖配置：

- `nodeFormKey`
- `nodeFormVersion`

生效优先级固定为：

- `节点表单 > 流程默认表单`

### 6.3 运行态使用规则

- 发起页永远使用流程默认表单
- 任务处理页优先使用节点覆盖表单
- 若节点未配置覆盖表单，则回退到流程默认表单
- 审批单详情页始终保留业务正文视图，不因节点覆盖表单而丢失原始业务数据展示

### 6.4 前端代码表单注册中心

前端维护静态注册中心，按以下维度映射真实 React 组件：

- `businessType`
- `formKey`
- `formVersion`
- `formType`
- `component`
- `supportedModes`

其中 `supportedModes` 至少支持：

- `start`
- `task`
- `detail`

## 7. 业务绑定模型

业务表与流程引擎表必须分离。

### 7.1 业务表

- `oa_leave_bill`
- `oa_expense_bill`
- `oa_common_request_bill`

### 7.2 平台绑定表

建议至少包含两张表：

#### `wf_business_process_binding`

字段建议：

- `businessType`
- `sceneCode`
- `processKey`
- `processDefinitionId`
- `enabled`
- `priority`

#### `wf_business_process_link`

字段建议：

- `businessType`
- `businessId`
- `processInstanceId`
- `processDefinitionId`
- `startUserId`
- `status`

### 7.3 发起链路

- 业务发起页先保存业务单据
- 根据 `businessType + sceneCode` 查绑定
- 找到当前生效流程定义
- 发起流程实例
- 写入 `wf_business_process_link`

## 8. 审批意见与实例事件

### 8.1 审批意见模型

审批意见不是单一文本。

每条意见至少包含：

- `actionType`
- `opinionText`
- `opinionTemplateCode`
- `attachments`
- `operatorUserId`
- `receiveTime`
- `readTime`
- `handleStartTime`
- `handleEndTime`
- `handleDuration`

### 8.2 审批意见配置

`审批意见配置` 仅负责：

- 快捷意见模板
- 操作按钮策略
- 意见工具栏配置

不负责实际意见记录。

### 8.3 实例事件

为了支持流程图动画回顾，后端应增加实例事件维度，而不是只依赖任务表拼装历史。

实例事件至少应覆盖：

- 节点进入
- 节点接收
- 节点已阅
- 节点处理开始
- 节点处理完成
- 认领
- 转办
- 退回
- 催办

## 9. 非目标

- 不做可视化表单设计器
- 不做独立表单管理后台
- 不为 `OA` 与 `流程管理` 各建设一套运行态详情页
- 不在本轮设计里把 AI Copilot、PLM 深度闭环一起塞进执行范围

## 10. 实施顺序

建议后续流程能力按以下顺序推进：

1. 文档与契约纠偏，废弃“独立表单管理”方向
2. 流程定义与设计器核心，补齐流程默认表单与节点覆盖表单选择
3. 业务流程绑定与 OA 业务发起闭环
4. 流程中心与审批单详情页闭环
5. 流程实例监控、操作日志、审批意见配置与高级平台能力
