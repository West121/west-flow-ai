# Phase 2 系统管理运营后台补齐设计

> 状态：Approved v1
> 日期：2026-03-22
> 阶段：Phase 2
> 适用范围：`codex/m0-foundation`

## 1. 目标

Phase 2 的目标不是继续扩展流程运行态，而是把平台运营后台补成一套完整、可维护、菜单清晰的系统管理能力，并清理 `shadcn-admin` 残留入口。

本阶段必须一次闭环以下内容：

- 字典管理
- 日志管理
- 监控管理
- 文件管理
- 通知管理
- 消息管理
- 残留路由与菜单清理
- 系统管理菜单结构统一

## 2. 当前现状

当前仓库已经具备以下系统管理基础：

- 用户、角色、菜单、公司、部门、岗位
- 代理关系、离职转办
- 通知渠道配置
- 触发器管理
- 审计拦截器基础能力
- MinIO 依赖与部分文件基础设施

当前也存在明显缺口：

- 没有字典管理后台
- 没有完整日志管理后台
- 没有完整监控管理后台
- 没有文件管理后台
- 通知只有“渠道配置”，没有模板、任务、发送记录的运营视角
- 没有站内消息管理后台
- 前端仍保留 `apps / chats / tasks / help-center` 无关残留路由
- 侧边栏系统管理存在重复菜单项

## 3. 总体方案

本阶段采用“平台运营后台一体化”方案：

- 后端新增系统管理相关领域与接口
- 前端新增对应独立中文 CRUD 页面
- 统一系统管理菜单树与权限点
- 同时删除无关残留路由与菜单

本阶段明确不做：

- 真运行态替换
- Flowable 真实运行态落地
- AI Copilot
- PLM 业务样例
- 高级审批能力继续扩展

## 4. 模块边界

### 4.1 字典管理

字典管理拆成两条资源线：

- 字典类型
- 字典项

能力要求：

- 独立列表、详情、新建、编辑页面
- 列表支持分页、模糊查询、筛选、排序、分组
- 字典项按所属字典类型管理
- 支持启用/停用状态

后端建议包：

- `com.westflow.system.dict.type`
- `com.westflow.system.dict.item`

### 4.2 日志管理

日志管理聚焦运营查询，不做删除：

- 审计日志
- 登录日志
- 通知发送日志
- 消息投递日志

能力要求：

- 列表页按时间、用户、模块、结果、类型查询
- 详情页展示请求摘要、操作结果和上下文
- 支持按时间范围和关键字过滤

后端建议包：

- `com.westflow.system.log.audit`
- `com.westflow.system.log.login`
- `com.westflow.system.log.notification`
- `com.westflow.system.log.message`

### 4.3 监控管理

监控管理聚焦平台内部作业状态与健康度：

- 编排扫描记录
- 触发执行记录
- 通知渠道健康状态
- 平台任务概览

能力要求：

- 提供列表与详情页
- 不做复杂图表平台，先做数据可查询、可追踪
- 与编排、通知、触发器域复用现有记录

后端建议包：

- `com.westflow.system.monitor`

### 4.4 文件管理

文件管理采用“MinIO 元数据后台”方案：

- 文件列表
- 文件详情
- 上传
- 下载
- 预览地址
- 逻辑删除

约束：

- 本阶段不做复杂文件夹系统
- 不做文档协同编辑
- 以文件元数据和对象存储地址为中心

后端建议包：

- `com.westflow.system.file`

### 4.5 通知管理

通知管理不是重复“通知渠道配置”，而是补齐运营视角：

- 通知模板
- 通知任务
- 通知发送记录
- 通知渠道配置并入通知管理菜单树

能力要求：

- 模板支持渠道类型、标题模板、内容模板、状态
- 发送记录支持按渠道、状态、目标、时间过滤
- 与现有 `notification` 域复用发送记录和渠道模型

后端建议包：

- `com.westflow.system.notification.template`
- `com.westflow.system.notification.record`

### 4.6 消息管理

消息管理指站内消息的运营后台：

- 消息列表
- 消息详情
- 消息状态
- 目标用户范围
- 投递记录

能力要求：

- 支持查看已读/未读
- 支持按用户、标题、发送状态、时间过滤
- 支持手工创建系统消息

后端建议包：

- `com.westflow.system.message`

## 5. 前端结构调整

系统管理菜单重构为以下层级：

- 组织管理
- 权限管理
- 代理与转办
- 字典管理
- 日志管理
- 监控管理
- 文件管理
- 通知管理
- 消息管理

页面规则继续遵守既有约束：

- 每个功能独立列表页
- 每个功能独立详情页
- 每个功能独立新建页
- 每个功能独立编辑页
- 中文菜单与中文页面文案
- 列表页统一支持分页、模糊查询、筛选、排序、分组

## 6. 清理项

本阶段必须删除或替换以下残留内容：

- `frontend/src/routes/_authenticated/apps`
- `frontend/src/routes/_authenticated/chats`
- `frontend/src/routes/_authenticated/tasks`
- `frontend/src/routes/_authenticated/help-center`

同时修正：

- `sidebar-data.ts` 中重复的“代理关系管理”
- `sidebar-data.ts` 中重复的“离职转办执行”
- 系统管理菜单分组不清晰的问题

## 7. 数据与接口策略

数据库迁移继续采用单文件策略：

- 只维护 `backend/src/main/resources/db/migration/V1__init.sql`

接口统一遵守：

- 路径前缀：`/api/v1/system/...`
- 分页资源统一支持 `PageRequest`
- 支持模糊查询、筛选、排序、分组
- 保持当前错误响应与权限校验风格不变

## 8. 权限策略

本阶段新增的系统管理模块默认只对平台管理员开放，后续再细化：

- 字典管理：`SYSTEM_ADMIN`
- 日志管理：`SYSTEM_ADMIN`
- 监控管理：`SYSTEM_ADMIN`
- 文件管理：`SYSTEM_ADMIN`
- 通知管理：`SYSTEM_ADMIN` 或 `PROCESS_ADMIN`
- 消息管理：`SYSTEM_ADMIN`

具体接口可先按 `SYSTEM_ADMIN` 收口，避免过早细分。

## 9. 验证要求

Phase 2 完成时，必须满足：

- 后端新增域都有控制器测试或服务测试
- 前端新增模块至少通过 `test / typecheck / lint / build`
- 路由残留被实际删除
- 侧边栏不再出现重复系统菜单
- `V1__init.sql` 能支持新模块种子数据与测试启动

## 10. 下一步执行

在本设计确认后，进入独立实施计划与多代理实现：

- 字典 + 消息
- 日志 + 监控
- 文件 + 通知
- 清理项 + 菜单统一 + 联调验证
