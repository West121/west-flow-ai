# AIBPMN 审批流程平台开发任务文档

> 状态：草案 v2
> 日期：2026-03-21
> 适用范围：`monorepo` 绿地项目，`frontend/` 直接二开 `shadcn-admin`，`backend/` 为单个 Spring Boot 应用并按领域分包，后续预留微服务拆分边界。

## 1. 项目目标

构建一个支持中国式复杂审批场景的 AIBPMN 审批流程平台，采用“双线推进”策略：

- 业务闭环线：优先跑通 `OA` 真实审批闭环，形成可演示、可试点、可验证的业务价值。
- 平台底座线：同步建设组织权限、流程定义、流程运行时、审计监控、通知提醒、AI 能力等可复用平台能力。

本阶段不追求一次性交付全部能力，而是按里程碑持续交付。首个业务锚点为 `OA`，后续扩展 `PLM`。

## 2. 范围与非目标

### 2.1 本文档覆盖范围

- 总体架构
- 仓库结构
- 前后端能力边界
- `M0-M4` 里程碑路线图
- 多 Agent 协同开发机制
- 联调、测试、验收门槛
- 风险与待确认事项

### 2.2 当前阶段非目标

- 不在第一阶段追求全部中国式审批能力一次性上线
- 不在前期拆分微服务部署，但代码结构必须为后续拆分预留边界
- 不做可视化表单设计器，流程表单与节点表单采用代码组件方式维护
- M2 仍不建设独立表单管理模块；流程默认表单与节点覆盖表单均在流程设计属性面板中配置，表单本体继续采用代码组件方式维护
- 不先做 PLM 深度业务闭环，PLM 作为平台泛化验证阶段进入

## 3. 总体架构

### 3.1 仓库结构

```text
/
├─ docs/                   # 规格、计划、联调文档、任务文档
├─ frontend/               # 基于 shadcn-admin 二开
├─ backend/                # 单个 Spring Boot 应用
├─ infra/                  # docker-compose、初始化脚本、部署模板
└─ scripts/                # 开发辅助脚本、校验脚本、生成脚本
```

### 3.2 前端技术方案

- 基线：`shadcn-admin`
- 构建：`Vite`
- 路由：`TanStack Router`
- 画布：`react-flow`
- 状态：`zustand`
- 服务端状态：`tanstack-query`
- 表单：`react-hook-form + zod`
- 请求：`axios`
- UI：`shadcn/ui`

### 3.3 后端技术方案

- 运行时：`Spring Boot`
- ORM：`MyBatis-Plus`
- 数据库：`PostgreSQL`
- 迁移：`Flyway`
- 缓存：`Redis`
- 对象存储：`MinIO`
- 权限：`Sa-Token`
- 流程引擎：`Flowable`
- 编排：`LiteFlow`
- 表达式：`Google Aviator`
- AI：`Spring AI + Spring AI Alibaba`

### 3.4 后端分包边界

后端保持单应用部署，但必须按领域包组织，而不是全局 `controller/service/mapper` 横切堆放。

```text
com.westflow
├─ common
├─ infra
├─ system
├─ identity
├─ organization
├─ file
├─ notification
├─ rule
├─ processdef
├─ processbinding
├─ processruntime
├─ processaction
├─ approval
├─ orchestrator
├─ analytics
├─ oa
└─ ai
```

说明：

- `processdef`：流程 DSL、版本、发布、DSL 转 BPMN
- `processbinding`：业务类型与流程定义绑定、业务单据与流程实例关联
- `processruntime`：实例、任务、轨迹、审批单详情、流程中心等运行时能力
- `processaction`：加签、减签、追加、认领、转办、委派、代理、拿回、唤醒等动作能力
- `approval`：审批意见记录、审批意见模板、审批工具栏与按钮配置
- `orchestrator`：触发器、定时器、超时处理、提醒调度、LiteFlow 编排
- `ai`：AI Copilot、流程设计智能体、智能填报、待办处理智能体、统计问答

### 3.5 业务与流程关系

业务表必须独立于流程引擎表，禁止把业务字段直接耦合进 Flowable 表结构。

建议关系模型：

- 业务主表：`oa_leave_bill`、`oa_expense_bill`、`oa_seal_bill` 等
- 表单数据表或表单 JSON：存业务提交流程所需数据快照
- 业务流程绑定表：记录 `businessType / sceneCode / processKey / processDefinitionId / enabled / priority`
- 业务流程关联表：记录 `businessType / businessId / processInstanceId / processDefinitionId / startUserId / status`
- 审批意见表：记录审批意见、附件、操作类型、意见来源、接收/阅读/办理时间
- 实例事件表：记录节点进入、接收、已阅、开始办理、办理完成、认领、转办、退回等事件

这样可确保：

- OA 与 PLM 复用同一流程平台
- 流程版本升级不污染业务模型
- 审批轨迹与业务状态可独立审计

### 3.6 流程管理产品结构

流程平台按“双入口、单中心、能力分层”设计。

#### OA

- 请假申请
- 报销申请
- 通用申请
- OA 流程查询

#### 流程管理

- 流程中心
- 流程定义
- 流程设计
- 流程版本
- 流程发布记录
- 流程实例监控
- 流程操作日志
- 审批意见配置
- 业务流程绑定

#### 流程中心

统一运行态中心，只保留一套实现：

- 待办
- 已办
- 我发起
- 抄送我
- 发起流程

规则：

- `OA > OA 流程查询` 进入同一套流程中心页面，但默认筛选 `businessDomain=OA`
- `流程管理 > 流程中心` 进入全域视角
- 不允许分别建设 OA 待办和流程管理待办两套页面

### 3.7 审批单详情页与表单策略

流程详情、实例详情、流程跟踪不再单独做菜单页，统一收敛到“审批单详情页”内部。

审批单详情页必须包含：

- 业务表单区
- 当前处理操作区
- 审批意见区
- 流程图预览区
- 流程回顾区
- 实例信息区
- 附件区
- 操作日志区

流程图预览与回顾要求：

- 内嵌在详情页，不单独跳转
- 支持已完成节点、当前节点、未到达节点、退回路径高亮
- 支持实例流程动画回顾
- 节点执行信息至少包含办理人、接收时间、阅读时间、办理时间、办理时长

表单策略要求：

- 不建设独立“流程表单管理”“节点表单管理”后台
- 流程定义根级配置 `processFormKey / processFormVersion`
- 审批节点可选配置 `nodeFormKey / nodeFormVersion`
- 生效优先级固定为 `节点表单 > 流程默认表单`
- 发起页永远使用流程默认表单
- 任务处理页在节点未配置表单时回退到流程默认表单
- 表单是写好的前端代码页面/组件，流程设计器属性面板只负责选择，不负责可视化设计

## 4. 前端强约束

以下规则为硬性规范，必须进入所有前端 Agent 的实现约束。

### 4.1 页面组织规则

- 左侧 `sidebar` 使用主子目录组织功能，不允许将多个 CRUD 子域塞进同一页面的多个页签中完成。
- 每个功能必须有独立路由与独立页面。
- 列表、详情、创建、编辑必须是独立页面。
- 轻量辅助操作允许使用 `Dialog`、`Sheet`、`Popover`，例如选人、审批意见、确认动作、上传附件。
- 流程设计器必须是独立工作区页面，不嵌入普通 CRUD 页面。

### 4.2 列表页统一能力

所有主列表页默认必须支持：

- 分页
- 模糊查询
- 精确筛选
- 分组
- 排序
- 列显隐
- 批量操作
- 导出预留
- 空状态
- 骨架屏
- 权限态控制

### 4.3 表单页统一能力

- 创建页与编辑页分离
- 支持“保存并返回列表”“保存并继续编辑”“取消返回列表”
- 表单校验必须在前端显式提示
- 服务端错误必须映射到字段级或页面级错误信息
- 表单字段必须预留后续 AI 智能填报集成点

### 4.4 查询协议约束

- 后端列表接口必须支持模糊查询，而不是仅支持精确匹配
- 后端列表接口需统一分页、排序、筛选、关键字搜索协议
- 复杂筛选需支持结构化条件对象，避免前端随意拼接 query 参数

### 4.5 设计器能力要求

- 节点拖拽
- 画布拖拽
- 节点位置拖动
- 自动布局
- redo / undo
- 辅助线
- 节点配置面板
- 流程配置面板
- 版本管理
- 校验提示
- 运行模拟预留

### 4.6 文案与语言规则

- 前端菜单、按钮、页面标题、表格列名、表单标签、状态文案默认全部使用中文。
- 本期不考虑 `i18n`，不做多语言抽象，不为英文切换预留额外实现成本。
- 仅技术字段名、代码标识符、接口字段名保持英文。

## 5. AI 能力强约束

- 前端只有一个统一 `AI Copilot` 入口，不单独再做第二套 AI 产品导航。
- 对话采用毛玻璃卡片式聊天布局。
- 对话消息必须支持结构化卡片，而不只是纯文本：
  - 表单预览卡片
  - 确认按钮卡片
  - 操作按钮卡片
  - 统计图卡片
  - 数据摘要卡片
- AI 对话必须持久化历史记录。
- AI 会话必须带业务上下文与用户上下文，避免上下文断裂。
- AI 功能必须受权限控制，用户只能调用自己权限范围内的 AI 能力。

## M3 平台监控与高级动作 1.0

### M3 当前冻结执行批次（2026-03-22 Advanced Runtime）

本批次先不做全部平台高级能力，而是先把“运行态的人和任务”做厚，形成更接近中国式审批的协作动作闭环。

#### 冻结范围

- `加签`
- `减签`
- `撤销`
- `催办`
- `真实抄送模型`
- `抄送已阅`
- `流程中心筛选器增强`
- `审批单详情动作轨迹增强`

#### 明确不在本批次内

- `驳回到发起人`
- `驳回到任意节点`
- `驳回策略`
- `跳转`
- `委派`
- `代理`
- `离职转办`
- `拿回`
- `唤醒`
- `追加`
- `动态构建`
- `定时`
- `触发`
- `自动提醒`
- `超时审批`
- `包容分支`

#### 设计边界

- 本批次继续复用现有 `processruntime` 平台层运行态模型，不引入第二套任务引擎。
- `抄送我` 必须从占位页升级为真实运行态任务列表，而不是仅保留菜单入口。
- 审批单详情页中的动作轨迹必须覆盖 `认领 / 转办 / 退回 / 加签 / 减签 / 撤销 / 催办 / 已阅`。
- 流程中心筛选必须从“基础分页检索”升级为“结构化运行态筛选”，支持业务类型、节点、实例状态、最新动作、已阅状态、时间区间等条件。

### M3 下一冻结执行批次（2026-03-22 Reject & Return Runtime）

本批次继续复用同一套 `processruntime`，把“驳回与回退策略”从零散动作提升为一套完整运行态语义。

#### 冻结范围

- `驳回到上一步人工节点`
- `驳回到发起人`
- `驳回到任意人工节点`
- `重新审批策略`
- `跳转`
- `拿回`
- `唤醒`
- `审批单详情动作轨迹继续补齐`

#### 设计边界

- 驳回必须拆成专用运行态动作，不再仅靠 `complete(action=REJECT)` 承载复杂语义。
- 本批次支持两种重审策略：
  - `CONTINUE`
  - `RETURN_TO_REJECTED_NODE`
- `跳转` 视为平台特权动作，本批次仅允许跳到人工审批节点或结束节点。
- `拿回` 仅允许在当前办理人尚未阅读、尚未开始办理时发生。
- `唤醒` 仅支持终态实例基于历史人工任务重新进入流程，不做 Flowable 引擎级回滚。
- 审批单详情页中的动作轨迹必须额外覆盖 `驳回 / 跳转 / 拿回 / 唤醒`，流程图回顾必须能高亮这些路径。

## 7. 多 Agent 协同开发机制

本文档默认后续使用多 Agent 并行开发。

### 7.1 Agent 分工原则

- 每个 Agent 必须有独立写入边界
- 优先按目录与领域切分，不按临时任务口头切分
- 公共契约优先冻结，再并行开发
- 公共组件、公共类型、接口 DTO 由指定 owner 负责

### 7.2 推荐 Agent 角色

- Agent A：前端平台与布局壳
- Agent B：前端工作台与 CRUD 页面范式
- Agent C：前端流程设计器
- Agent D：后端基础设施与权限
- Agent E：后端流程定义与 DSL 转 BPMN
- Agent F：后端流程运行时与动作能力
- Agent G：后端 OA 业务域
- Agent H：AI 与 Copilot
- Agent I：测试、联调、回归、契约校验

### 7.3 并行开发边界

#### 可并行

- `frontend/` 页面壳与 `backend/` 基础设施
- `processdef` 与 `processruntime`
- `oa` 业务域与 `ai` 模块
- 表格基座与 CRUD 业务页面

#### 必须串行或先冻结契约

- 通用分页与查询协议
- 用户上下文与权限模型
- React Flow DSL 结构定义
- DSL 转 BPMN 映射契约
- 审批动作枚举与状态机
- AI 工具调用协议

### 7.4 分支与合并建议

- 主分支：`main`
- 功能分支：`feature/<milestone>-<scope>`
- 公共契约类改动先合并
- Agent 分支禁止同时改动同一公共文件，除非已指定 owner
- 合并前必须过最小验证集

### 7.5 文档驱动要求

所有 Agent 在编码前必须先读取：

- 本文档
- 当前里程碑任务拆分文档
- API 契约文档
- DSL 结构文档
- 测试基线文档

### 7.6 共享契约注册表

以下共享契约必须先冻结 owner、来源文件、变更规则，再允许多 Agent 并行：

| 契约 | owner | 来源文件/目录 | 冻结时点 | 变更规则 |
| --- | --- | --- | --- | --- |
| 分页查询协议 | 后端基础设施 owner | `docs/contracts/pagination.md` `backend/.../common/query` | M0 | 需前后端 owner 双确认 |
| 统一错误码 | 后端基础设施 owner | `docs/contracts/errors.md` `backend/.../common/error` | M0 | 需前后端 owner 双确认 |
| 用户上下文与权限模型 | 后端权限 owner | `docs/contracts/auth.md` `backend/.../identity` | M0 | 需后端与 AI owner 双确认 |
| React Flow DSL | 流程定义 owner | `docs/contracts/process-dsl.md` `frontend/.../dsl` `backend/.../processdef` | M0 | 需前端设计器与流程定义 owner 双确认 |
| DSL 转 BPMN 映射规则 | 流程定义 owner | `docs/contracts/dsl-bpmn-mapping.md` | M1 | 需流程定义与运行时 owner 双确认 |
| 审批动作枚举与状态机 | 流程运行时 owner | `docs/contracts/task-actions.md` | M1 | 需运行时与 OA owner 双确认 |
| 通用表格查询字段协议 | 前端 CRUD owner | `docs/contracts/table-query.md` `frontend/.../shared/table` | M0 | 需前端 CRUD 与后端基础设施 owner 双确认 |
| AI 工具调用协议 | AI owner | `docs/contracts/ai-tools.md` `backend/.../ai` | M2 | 需 AI、流程、OA owner 三方确认 |

执行要求：

- 未登记到共享契约注册表的公共接口，不允许作为多 Agent 并行依赖。
- 共享契约变更必须先更新文档，再改代码。
- 契约变更必须在变更说明中写明影响范围、升级策略、兼容性结论。
- 如变更会影响两个及以上 Agent 的写入边界，必须先合并契约分支，再继续业务实现。

### 7.7 多 Agent 合并与冲突处理规则

- 所有 Agent 必须先以契约分支或基础设施分支完成公共基线，再切业务分支。
- 公共文件 owner 机制必须显式指定：
  - `frontend/src/routes` 由前端平台 owner 负责
  - `frontend/src/features/shared/table` 由前端 CRUD owner 负责
  - `frontend/src/features/designer/dsl` 由设计器 owner 负责
  - `backend/.../common` 与 `backend/.../infra` 由后端基础设施 owner 负责
  - `backend/.../processdef` 由流程定义 owner 负责
  - `backend/.../processruntime` 与 `backend/.../processaction` 由流程运行时 owner 负责
  - `backend/.../ai` 由 AI owner 负责
- 非 owner 如需改动共享文件，必须先提交变更申请并获得 owner 确认。
- 合并顺序固定为：契约 -> 公共基础设施 -> 平台能力 -> 业务实现 -> AI 集成 -> 联调修复。
- 联调修复不得直接绕过契约文档修改公共接口。

## 8. 联调策略

### 8.1 联调基本原则

- 先契约，后并行
- 先 Mock 跑通页面，再接真实接口
- 关键接口冻结后方可大规模并行开发
- 联调问题必须回写到文档或缺陷清单

### 8.2 联调阶段划分

#### 阶段 A：前端自联调

- 前端基于 Mock 数据完成列表、详情、创建、编辑、流程处理页
- 设计器基于本地 DSL 示例完成画布交互

#### 阶段 B：接口契约联调

- 对接统一分页协议
- 对接统一错误码与权限异常
- 对接流程发起、待办列表、任务处理、轨迹查询

#### 阶段 C：流程闭环联调

- 业务提交 -> 流程发起
- 待办处理 -> 实例推进
- 驳回/退回/撤销/催办/转办/委派
- 轨迹、意见、附件、通知全链路校验

#### 阶段 D：AI 联调

- AI 获取上下文
- AI 触发表单预填
- AI 触发流程操作
- AI 统计查询与权限校验

### 8.3 环境矩阵

- 本地开发环境
- 集成联调环境
- 测试环境
- 预发布环境

### 8.4 测试数据与环境责任矩阵

#### 测试种子数据

- 必备组织数据：公司、部门、岗位、用户、角色、兼任、代理关系
- 必备流程数据：请假、报销、通用申请流程定义与版本
- 必备业务数据：草稿单、审批中单据、已完成单据、已撤销单据
- 必备权限数据：管理员、流程设计者、普通员工、部门负责人、财务、抄送对象

#### 环境责任分配

| 环境 | 责任 owner | 主要用途 | 必备内容 |
| --- | --- | --- | --- |
| 本地开发 | 各自 Agent | 单模块开发与自测 | 本地依赖、最小种子数据 |
| 集成联调 | 测试/联调 owner | 前后端、流程、AI 联调 | 稳定接口、统一测试数据、联调日志 |
| 测试环境 | QA owner | 回归、验收、性能预检 | 完整业务链路数据、自动化测试报告 |
| 预发布环境 | 发布 owner | 上线前验收 | 接近生产配置、审计与监控开启 |

#### 契约测试要求

- 后端需提供接口契约样例与错误场景样例
- 前端需基于契约样例维护 Mock 数据
- 流程 DSL 需提供标准样例与反例
- AI 工具调用需提供工具输入输出样例
- 所有样例必须纳入版本控制

| M3 | AI 三个场景验收、AI 权限校验、上下文连续性验证、工具调用审计验证 | 测试 + 预发布 | AI 可越权操作；无审计；无法基于业务上下文连续工作 |

#### 合并前最小检查集

- 前端：`lint + typecheck + 单测 + 受影响页面冒烟`
- 后端：`单测 + 集成测试 + 受影响流程动作测试`
- 流程：`DSL 样例转换测试 + 核心流程回归`
- AI：`工具调用用例 + 权限校验 + 审计校验`

#### 交付声明要求

任一 Agent 声称“完成”时，必须同时给出：

- 改动范围
- 受影响契约
- 已执行验证项
- 未覆盖风险
- 联调依赖项
