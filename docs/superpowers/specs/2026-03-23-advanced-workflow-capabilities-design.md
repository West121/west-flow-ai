# 高级流程能力总设计

> 状态：Approved v1
> 日期：2026-03-23
> 适用范围：`main` 后续高级流程能力阶段

## 1. 背景

当前平台已经完成：

- 真实 `Flowable BPMN` 运行态
- 流程定义、设计器、发布、流程管理后台
- 流程中心、审批单详情、流程轨迹与流程图回顾
- 加签、减签、转办、委派、代理、离职转办、抄送、已阅、撤销、催办
- 驳回、退回、跳转、拿回、唤醒
- 顺序会签、并行会签、或签、票签
- 自动化基础：超时审批、自动提醒、定时节点、触发节点

但剩余高级流程能力仍缺统一方案。当前最主要的问题不是“单点功能没做”，而是如果继续逐块追加，后面很容易把 DSL、BPMN 转换、运行态接口、轨迹模型再次打散。

因此这一轮先冻结一份总设计文档，把：

- 已完成能力的真实映射口径
- 本轮先开发的能力
- 后续能力的统一实现边界

一次性写清。

## 2. 目标

本总设计只做三件事：

1. 统一高级流程能力分层，避免后续各批次各自发明模型。
2. 明确每类能力在 `设计器 -> DSL -> BPMN -> 运行态 -> 轨迹/详情` 的落点。
3. 为后续 `主子流程 / 终止高级策略 / 追加 / 动态构建 / 包容分支 / 沟通 / 穿越时空` 预留稳定扩展点。

## 3. 能力分层

### 3.1 已完成能力

这一层已经进入真实运行态，不属于下一批新增功能：

- 任务动作类
  - 认领
  - 转办
  - 驳回
  - 退回
  - 跳转
  - 拿回
  - 唤醒
  - 撤销
  - 催办
  - 已阅
- 协同类
  - 加签
  - 减签
  - 抄送
  - 委派
  - 代理
  - 离职转办
- 审批策略类
  - 顺序会签
  - 并行会签
  - 或签
  - 票签
- 自动化类
  - 超时审批
  - 自动提醒
  - 定时节点
  - 触发节点

这些能力后续只允许在现有真实 `Flowable` 口径上增强，不再重做第二套实现。

### 3.2 本轮优先开发

本轮只开发：

- 主子流程
- 终止高级策略

这两项的共同特征是：

- 会影响流程结构
- 会影响父子实例关系
- 会影响轨迹和详情建模

因此它们必须优先做成统一底座。

### 3.3 后续高级能力

本总设计同时覆盖这些能力，但不在本轮开发：

- 追加
- 动态构建
- 包容分支
- 沟通
- 穿越时空

这些能力后续必须复用本轮形成的：

- 结构节点扩展方式
- 父子实例/附属实例关联表
- 平台事件轨迹模型
- 终止与回收策略

## 4. 统一设计原则

### 4.1 Flowable 优先，平台补语义

高级流程能力的总原则固定为：

- 能直接映射到 `Flowable` 原生能力的，优先使用原生能力
- `Flowable` 不直接承载的平台产品语义，由平台表和平台事件补足
- 不允许为“看起来方便”再发明第二套执行引擎

### 4.2 结构能力和动作能力分层

- `结构能力`
  - 子流程
  - 动态构建
  - 包容分支
  - 追加
- `动作能力`
  - 终止
  - 沟通
  - 穿越时空
  - 既有加签减签/跳转/驳回等

结构能力主要落在：

- 设计器节点
- DSL 字段
- BPMN 转换
- 实例关联表

动作能力主要落在：

- 运行态服务
- 平台事件表
- 审批详情轨迹
- 权限和确认策略

### 4.3 所有高级能力都必须闭环

每项高级能力都必须具备以下闭环：

- 设计器可配置
- DSL 可校验
- BPMN 或运行态可映射
- 运行态可执行
- 审批详情可展示
- 轨迹可留痕
- 实例监控可查询

## 5. 主子流程总方案

### 5.1 节点模型

新增 `subprocess` 节点类型，核心字段：

- `calledProcessKey`
- `calledVersionPolicy`
  - `LATEST_PUBLISHED`
  - `FIXED_VERSION`
- `calledVersion`
- `businessBindingMode`
  - `INHERIT_PARENT`
  - `OVERRIDE`
- `terminatePolicy`
  - `TERMINATE_SUBPROCESS_ONLY`
  - `TERMINATE_PARENT_AND_SUBPROCESS`
- `childFinishPolicy`
  - `RETURN_TO_PARENT`
  - `TERMINATE_PARENT`
- `inputMappings`
- `outputMappings`

### 5.2 Flowable 映射

主子流程统一基于 `callActivity` 实现：

- 父流程节点映射为 `callActivity`
- 子流程必须是已发布流程定义
- 版本策略由平台在发布/部署时解析
- 输入输出变量映射由平台统一收口，不允许分散在控制器里做

### 5.3 平台补充表

新增 `wf_process_link`：

- `root_instance_id`
- `parent_instance_id`
- `child_instance_id`
- `parent_node_id`
- `called_process_key`
- `called_definition_id`
- `link_type`
- `status`
- `terminate_policy`
- `child_finish_policy`
- `created_at`
- `finished_at`

这张表是后续主子流程、追加、动态构建的统一结构关系底座。

## 6. 终止高级策略总方案

### 6.1 终止范围

统一定义三种终止范围：

- `CURRENT`
  - 终止当前流程实例
- `CHILD`
  - 终止当前子流程实例
- `ROOT`
  - 终止整条根流程及其所有子流程

### 6.2 终止策略

统一定义两类策略：

- `SOFT_TERMINATE`
  - 终止实例并记录原因、发起人、作用域
- `CASCADE_TERMINATE`
  - 终止当前实例，同时级联终止其所有子实例

### 6.3 平台事件

终止事件必须至少记录：

- `scopeType`
  - `PROCESS_INSTANCE`
  - `SUB_PROCESS_INSTANCE`
- `terminateScope`
  - `CURRENT`
  - `CHILD`
  - `ROOT`
- `terminateReason`
- `terminatedBy`
- `terminatedAt`

这些字段进入统一审批轨迹和实例监控。

## 7. 后续高级能力预留方案

### 7.1 追加

追加不单独新起一套实例模型，统一视为：

- 在现有实例树下新增附属节点任务或附属子流程
- 复用 `wf_process_link` 记录附属关系

### 7.2 动态构建

动态构建统一定义为：

- 运行时根据规则创建额外审批节点或子流程调用
- 复用本轮的结构扩展点
- 不允许在控制器里临时拼流程图语义

### 7.3 包容分支

包容分支后续统一映射到：

- 设计器独立节点类型
- BPMN `inclusive gateway`
- 平台轨迹补充分支命中信息

### 7.4 沟通

沟通不作为普通评论处理，后续统一建模为：

- 运行态协同事件
- 不改变主任务所有权
- 在审批详情和轨迹中单独显示

### 7.5 穿越时空

穿越时空后续统一定义为高级回退/重开策略，不在当前动作里偷塞语义。

## 8. 前端统一交互原则

### 8.1 设计器

设计器里所有高级能力都必须以节点配置形式出现：

- 不做隐藏式运行时黑盒逻辑
- 必须在节点面板可见、可保存、可回显

### 8.2 审批详情

审批详情统一承担三类展示：

- 主流程与子流程结构
- 高级动作轨迹
- 结构关系与终止结果

### 8.3 实例监控

实例监控统一承担：

- 主子流程树
- 终止范围
- 终止原因
- 子流程完成/终止联动结果

## 9. 代码分层规划

建议保持以下分层不变：

- `processdef`
  - DSL 模型、校验、BPMN 转换
- `flowable`
  - Flowable helper / facade
- `processruntime`
  - 运行态查询、动作、详情组装
- `orchestrator`
  - 自动化动作
- `workflowadmin`
  - 流程管理后台

新增能力按职责落点：

- 主子流程
  - `processdef`
  - `processruntime`
  - `flowable`
- 终止高级策略
  - `processruntime`
  - `flowable`
  - 轨迹/监控查询

## 10. 当前开发顺序

按后续顺序，建议这样推进：

1. 主子流程 + 终止高级策略
2. 追加 + 动态构建
3. 包容分支
4. 沟通 + 穿越时空

## 11. 非目标

本总设计明确不在当前文档中解决：

- AI 审批策略
- OA/PLM 业务表单细节
- 新的组织权限模型

这些继续按各自专题文档推进，不混进高级流程结构设计。
