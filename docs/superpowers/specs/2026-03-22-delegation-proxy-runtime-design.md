# 委派与代理运行态设计

> 状态：Frozen v1
> Owner：流程运行时 owner
> 冻结时点：2026-03-22

## 目标

在现有 `processruntime` demo 模型内补齐以下运行态能力：

- `委派`
- `代理`
- `离职转办`
- `转办链路增强`

本批次只做运行态闭环，不做系统管理配置页面。

## 术语与语义

### 转办

- 转办仍保持现有语义。
- 当前办理人把当前任务完全转给目标用户。
- 原任务进入 `TRANSFERRED`，新任务进入 `PENDING`。

### 委派

- 委派是任务级动作。
- 当前办理人把当前任务委派给目标用户代办，但必须保留原责任人与审计关系。
- 原任务进入 `DELEGATED`。
- 系统创建新的委派任务，目标用户处理该委派任务。
- 审批单详情和动作轨迹必须能看到“谁委派给谁”。

### 代理

- 代理是预设关系，来源于当前登录态中的 `delegations`。
- 本批次仅覆盖“已分配的待处理任务”，不覆盖公共任务认领、抄送、流程管理员跳转。
- 若 `A -> B` 为激活代理关系，则分配给 `A` 的 `PENDING` 人工任务，`B` 可代为处理。
- `B` 代办后：
  - 任务实际处理人记录为 `B`
  - 任务责任人仍保留为 `A`
  - `A` 与 `B` 都应能在查询链路中看到该历史任务
- 若 `A` 先自行处理，则 `B` 不再看到该待办。

### 离职转办

- 离职转办是平台批量动作。
- 仅流程管理员可执行。
- 本批次仅处理当前活跃的 `PENDING` 人工任务，不处理：
  - `PENDING_CLAIM`
  - `CC`
  - 已完成历史任务
- 每个被转办的任务都必须留下单独动作轨迹。
- 原任务进入 `HANDOVERED`，新任务进入 `PENDING`。

## 设计边界

- 不新增系统管理 CRUD 页面。
- 不新增代理关系配置页。
- 继续复用 `FixtureAuthService` 中的代理关系种子数据。
- 流程管理员识别继续复用现有角色模型，`PROCESS_ADMIN` 视为可执行离职转办。
- 本批次不做：
  - 主办
  - 自动代理生效时间窗配置
  - 委派撤回
  - 离职转办批处理任务中心

## 后端变更

### 新动作

- `POST /api/v1/process-runtime/demo/tasks/{taskId}/delegate`
- `POST /api/v1/process-runtime/demo/users/{sourceUserId}/handover`

### 新状态

- `DELEGATED`
- `HANDOVERED`

### 新审计字段

任务轨迹和实例事件需要补充以下信息：

- `actingMode`
  - `DIRECT`
  - `PROXY`
  - `DELEGATE`
  - `HANDOVER`
- `actingForUserId`
- `delegatedByUserId`
- `handoverFromUserId`

### 可见性规则

- `TODO`
  - 直接办理人可见
  - 激活代理关系中的代理人也可见
- `DONE`
  - 直接处理人可见
  - 若为代理代办，则责任人和代理人都可见
- `审批单详情`
  - 任一拥有当前列表可见性的用户都可打开

## 前端变更

### 流程中心

- 审批单详情页右侧动作区新增：
  - `委派`
- 待办列表与详情摘要补充：
  - `代理代办`
  - `委派任务`
  - `离职转办`

### 全局批量动作

- 在流程中心 `待办` 页工具栏增加“离职转办”入口。
- 本批次使用简单对话框采集：
  - 来源用户 ID
  - 目标用户 ID
  - 说明

### 动作轨迹

- 动作轨迹文案新增：
  - `委派`
  - `代理代办`
  - `离职转办`
- 若存在 `actingForUserId`，详情中必须明确展示“代谁办理”。

## 验收标准

- 当前办理人可对任务执行委派。
- 代理人可以在不改配置后台的情况下处理被代理人的已分配任务。
- 代理代办后，责任人和代理人都能在查询链路中追踪到该任务。
- 流程管理员可对某个来源用户执行离职转办，批量生成新待办。
- 审批单详情、动作轨迹、流程中心列表文案与状态一致。
