# Phase 7：平台收口与生产化深化设计

> 状态：Draft
> 日期：2026-03-23
> 适用范围：`main` 分支在 `AI + PLM` 基础闭环后的整体剩余任务

## 1. 目标

在当前基线下，不再新增大而散的产品面，而是优先收掉真实生产化落差、历史命名债、运行态轨迹缺口和通知渠道落地问题；随后再进入 AI / PLM 深化和前端工程化优化。

本阶段的核心要求是：

- 先解决“当前已经能跑，但仍有历史过渡痕迹和生产化缺口”的问题
- 再推进 AI / PLM 的深水区能力
- 继续坚持并行，但每条线必须边界清晰

## 2. 当前真实剩余任务

结合当前代码、契约和最近提交，剩余任务分为两类。

### 2.1 必须优先收口的任务

- 清理运行态历史命名与旧接口残留
- 将实例事件、自动化轨迹、通知发送轨迹真正接到真实历史链路
- 将短信、企业微信、钉钉通知从 mock 升级到真实生产适配器
- 统一刷新过期契约与路线图文档，移除 demo 时代描述

### 2.2 深化类任务

- AI Copilot 继续做深：设计智能体、复杂待办解释、更多可执行工具
- PLM 域继续做深：更完整的业务对象、发布/BOM/变更链路
- 前端工程化收尾：warning 清理、chunk 优化、E2E 回归

## 3. 总体分期

### Phase 7A：历史债清理与运行态收口

目标：

- 去掉仓库中“已经不是 demo、但命名和契约仍保留 demo 语义”的残留
- 把真实 Flowable 的轨迹存储和自动化历史查询补齐

包括：

- `DemoTaskView` 重命名与对外契约收口
- `/process-runtime/demo/*` 文档和前端残留引用清理
- `FlowableProcessRuntimeTraceStore` 真正落实现有查询接口
- 审批单详情、流程回放、自动化轨迹改为优先使用真实历史/平台事件

### Phase 7B：通知渠道生产化

目标：

- 保留现有 `IN_APP / EMAIL / WEBHOOK`
- 把 `SMS / WECHAT / DINGTALK` 从 mock 适配器升级到真实可配置适配器

包括：

- 渠道配置字段补齐
- Provider 实现与错误处理
- 发送审计和失败重试
- 诊断页状态联动

### Phase 7C：契约与文档重整

目标：

- 让文档反映当前真实代码状态，而不是保留阶段性历史说明

包括：

- 更新总设计文档
- 更新任务动作契约
- 更新 AI / PLM / 认证 / 运行态契约
- 废弃过时的 demo 相关说明

### Phase 7D：AI / PLM 深化

目标：

- 在当前 AI + PLM 闭环基础上，继续把业务能力做厚

包括：

- AI 流程设计智能体深化
- AI 复杂待办处理与解释
- PLM 复杂对象与流程链路扩展
- AI 与 PLM 的统计分析深化

### Phase 7E：前端工程化与性能收尾

目标：

- 清理历史 warning
- 降低大 chunk
- 补关键 E2E 回归

## 4. 并行原则

### 4.1 第一批允许并行的 3 条线

第一批可立即并行推进：

- Lane A：运行态历史债清理与轨迹落地
- Lane B：通知渠道生产化
- Lane C：文档与契约重整

这三条线写集边界明确：

- Lane A 主要写 `processruntime / oa / plm / frontend workbench & oa`
- Lane B 主要写 `notification / aiadmin diagnostics`
- Lane C 主要写 `docs/**`

### 4.2 第二批并行线

待第一批稳定后，再并行：

- Lane D：AI 深化
- Lane E：PLM 深化
- Lane F：前端工程化收尾

原因：

- Lane D / E 会消费第一批收口后的真实历史、真实通知、更新后的契约
- Lane F 依赖前两批接口和页面结构稳定

## 5. 第一批详细定义

### 5.1 Lane A：运行态历史债清理与轨迹落地

目标：

- 去掉 `DemoTaskView` 等历史命名
- 去掉前端和契约中的 `/process-runtime/demo/*`
- 让 `FlowableProcessRuntimeTraceStore` 不再是空骨架

必须具备：

- 新的任务视图命名和响应类型
- 实例事件查询有真实数据来源
- 自动化轨迹和通知记录查询不再固定返回空列表

不做：

- 扩展新审批动作
- 大规模改写 Flowable 主链路

### 5.2 Lane B：通知渠道生产化

目标：

- 企业微信、钉钉、短信具备真实发送实现

必须具备：

- 配置化 endpoint / credential
- 失败错误码回写
- 诊断页和发送记录可见

不做：

- 新增更多通知渠道
- 复杂模板编排引擎

### 5.3 Lane C：文档与契约重整

目标：

- 文档对齐当前代码状态

必须具备：

- 删除或标记废弃的 demo 文档描述
- 刷新流程动作契约
- 刷新认证、AI 富响应、PLM 助手契约

不做：

- 新产品设计
- 新路线图发散

## 6. 验收标准

### Lane A 验收

- 仓库中核心运行态不再暴露 `demo` 命名残留
- 审批单详情和轨迹查询具备真实历史来源

### Lane B 验收

- `SMS / WECHAT / DINGTALK` 至少具备真实 provider 实现
- 发送失败能在日志和诊断页看到

### Lane C 验收

- 主要契约与总路线图和当前代码对齐
- 旧 demo 描述被清理或显式废弃

## 7. 当前推荐下一步

直接按本设计启动第一批 3 条并行线：

- Lane A：运行态历史债清理与轨迹落地
- Lane B：通知渠道生产化
- Lane C：文档与契约重整

同时由主线程负责：

- 最终集成
- 全量验证
- 下一批 AI / PLM 深化分解
