# Phase 7：平台收口与生产化深化设计

> 状态：Current v2026-03-23
> 日期：2026-03-23
> 适用范围：`main` 分支在基础闭环完成后的当前剩余任务

## 1. 当前起点

本设计建立在以下真实状态之上：

- 流程运行时已经切到真实 `Flowable`
- 登录认证已经切到真实数据库账号体系
- `OA` 主链路已经具备真实发起、待办、审批单、任务动作闭环
- `AI Copilot` 已完成基础对话、工具路由、写确认、富响应块闭环
- `PLM` 已完成 `ECR / ECO / 物料主数据变更` 的业务与审批联查基础闭环

因此，Phase 7 不是“补基础设施”，而是“清历史债、补生产化缺口、继续做深业务能力”。

## 2. 当前阶段目标

Phase 7 聚焦三类工作：

- 收掉 demo 时代遗留的命名、文档和轨迹空洞
- 把通知、历史查询、审计信息补到真实生产可用口径
- 在稳定基线上继续深化 AI 与 PLM，而不是重新铺基础面

## 3. 当前剩余任务分组

### 3.1 收口类任务

- 清理运行态历史命名与过时 demo 描述
- 补齐实例事件、自动化轨迹、通知发送轨迹
- 刷新契约与文档，统一到当前真实代码状态

### 3.2 生产化类任务

- 将短信、企业微信、钉钉从 mock 适配器升级为真实可配置 provider
- 完善发送审计、诊断信息和失败重试语义

### 3.3 深化类任务

- AI Copilot 深化：更复杂的任务解释、流程设计辅助、更多业务工具
- PLM 深化：更丰富的业务对象、变更链路、统计分析
- 前端工程化收尾：warning 清理、chunk 优化、关键回归

## 4. 当前并行车道

### Lane A：运行态历史债清理与轨迹落地

目标：

- 去掉对外契约中的 demo 运行态口径
- 补齐真实历史查询和事件轨迹来源

重点：

- 核心视图命名收口
- 审批单详情、流程回放、自动化轨迹优先走真实历史链路
- 历史残留类名允许短期存在，但不能继续映射为“demo 运行时”

### Lane B：通知渠道生产化

目标：

- `SMS / WECHAT / DINGTALK` 具备真实发送能力与诊断结果

重点：

- 渠道配置字段
- provider 实现
- 失败回写与可观测性

### Lane C：文档与契约重整

目标：

- 让总设计、路线图和主契约全部反映当前真实状态

重点：

- 显式废弃 `/process-runtime/demo/*` 和 `ProcessDemoService` 的旧口径
- 刷新 `task-actions`、`auth`、`ai-rich-response`、`plm-assistant-tools`
- 验证契约文档完整性

### Lane D：AI 深化

目标：

- 在现有 Copilot 基线上继续增强任务处理、结果语义和业务摘要能力

### Lane E：PLM 深化

目标：

- 在现有 `ECR / ECO / 物料主数据变更` 基线上继续完善业务深度

### Lane F：前端工程化收尾

目标：

- 清 warning、降 chunk、补关键回归，不改变当前已稳定的业务闭环

## 5. 当前验收口径

### 平台收口验收

- 主要文档和契约不再把 demo 口径当作当前状态
- 主要运行态查询接口按真实 `Flowable` 运行时描述

### 生产化验收

- 通知渠道具备真实 provider 或明确生产化适配方案
- 历史轨迹、发送日志、失败诊断具备真实来源

### 深化验收

- AI 输出具备更完整的业务语义
- PLM 能提供更业务化的查询与摘要结果
