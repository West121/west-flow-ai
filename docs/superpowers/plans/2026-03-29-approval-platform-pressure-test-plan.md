# 审批平台压测计划

## 目标

验证当前审批平台在中高并发下的稳定性、延迟、错误率和关键依赖资源使用情况，重点覆盖：

- 待办/已办/我发起列表查询
- 审批详情加载
- 单任务办理动作
- 批量审批动作
- 会签 / 加签 / 驳回 / 退回等复杂动作
- 协同已阅
- SLA 提醒与升级执行链

## 压测范围

### 核心读场景

- 待办列表分页查询
- 审批单详情查询
- 审批轨迹 / 时间轴查询
- 流程图回顾数据查询

### 核心写场景

- 认领
- 同意
- 驳回
- 退回
- 加签
- 减签
- 转办
- 委派
- 撤销
- 批量认领
- 批量同意
- 批量驳回
- 协同已阅

### 调度场景

- SLA 提醒扫描
- SLA 升级扫描
- Orchestrator 事件处理

## 数据准备

- 至少 200 个审批实例
- 至少 1000 条任务记录
- 至少覆盖：
  - 普通单人审批
  - 串签
  - 并签
  - 抄送 / 督办 / 会办 / 阅办 / 传阅
  - 加签任务
  - 已撤销 / 已终止实例
- 至少 20 个可用于批量动作的同类任务

## 执行工具建议

- 首选：`k6`
- 备选：`JMeter`

原因：

- `k6` 更适合版本化脚本管理
- 可直接接入 CI 和阈值断言
- 对 REST 场景足够

## 并发阶段

### Stage 1：基线

- 并发用户：10
- 持续时间：5 分钟
- 目标：验证脚本、鉴权、数据准备无误

### Stage 2：中等负载

- 并发用户：50
- 持续时间：10 分钟
- 目标：观察日常规模下接口稳定性

### Stage 3：高负载

- 并发用户：100
- 持续时间：15 分钟
- 目标：验证主要读写链路的 p95 和错误率

### Stage 4：峰值冲击

- 并发用户：200
- 持续时间：10 分钟
- 目标：发现连接池、数据库锁、Flowable 执行瓶颈

## 指标

### 接口指标

- 成功率
- 错误率
- p50 / p95 / p99 延迟
- 超时比例

### 系统指标

- JVM 堆使用率
- 线程池活跃数
- 数据库连接池使用率
- PostgreSQL CPU / 内存 / 慢 SQL
- Flowable job backlog
- 任务表、轨迹表写入热点

### 业务指标

- 审批动作成功数
- 批量动作部分失败数
- 加签 / 驳回 / 退回后的轨迹完整性
- SLA 事件触发数与延迟

## 通过标准

### 读场景

- p95 小于 1000ms
- 错误率小于 1%

### 单任务写场景

- p95 小于 1500ms
- 错误率小于 1%

### 批量动作

- p95 小于 3000ms
- 错误率小于 2%

### 调度场景

- 无 backlog 持续堆积
- 无重复触发
- 无死锁或长事务

## 风险点

- 会签和加签组合场景可能导致任务组重建压力增大
- 批量动作可能放大 Flowable runtime 与 trace store 写压力
- SLA 扫描和人工操作并发时可能放大锁竞争
- 审批详情聚合查询可能在大轨迹数据下退化

## 首轮压测建议顺序

1. 待办列表 + 审批详情只读压测
2. 单任务认领 / 同意 / 驳回
3. 加签 / 退回 / 转办组合写压测
4. 批量审批
5. 协同已阅
6. SLA 提醒 / 升级

## 产出物

- 每轮压测报告：
  - 参数
  - 结果摘要
  - 瓶颈分析
  - 改进建议
